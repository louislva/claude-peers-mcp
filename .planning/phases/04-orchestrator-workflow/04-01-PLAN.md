async function brokerFetch<T>(path: string, body: unknown): Promise<T> {
  const res = await fetch(`${BROKER_URL}${path}`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(body),
  });
  if (!res.ok) {
    const err = await res.text();
    throw new Error(`Broker error (${path}): ${res.status} ${err}`);
  }
  return res.json() as Promise<T>;
}

**2. `export async function discoverPeers(myId: PeerId, gitRoot: string): Promise<{ proxy: AvailablePeer | null; executors: AvailablePeer[] }>`**
- Call `brokerFetch<PeerAvailabilityResponse>("/peer-availability", { repo: gitRoot, exclude_id: myId })`
- Merge candidates: `[...result.repo_peers.available, ...result.machine_peers.available]`
- ORCH-02: Find proxy by `candidate.summary.toLowerCase().includes("decision proxy")` — at most one; take the first match
- All other candidates are executors
- Return `{ proxy, executors }`

**3. `export interface PhaseNode { number: number; name: string; dir: string; dependencies: number[]; status: string; filesModified: string[] }`**

**4. `export function parseRoadmapPhases(roadmapContent: string): PhaseNode[]`**
- Parse ROADMAP.md text to extract phase entries
- For each phase: extract number (from "Phase N:" or "Phase N.N:"), name, status (completed/not started)
- Parse `**Depends on**:` field for explicit dependencies — look for patterns like "Phase 1", "Phase 2, Phase 3", "Nothing"
- Parse `**Goal**:` field and store in `name`
- Extract phase directory from plan list entries (e.g., `01-01-PLAN.md` -> dir is `01-foundation` from the phase heading)
- If a phase is marked `[x]` (completed), set `status: "completed"`; if `[ ]`, set `status: "pending"`
- Return array of PhaseNode objects

**5. `export function buildExecutionWaves(phases: PhaseNode[]): PhaseNode[][]`**
- Filter OUT completed phases — only schedule pending phases
- Kahn's algorithm (BFS topological sort):
  - Build inDegree map: for each pending phase, count how many of its dependencies are also pending (completed deps don't count — they're already satisfied)
  - Build dependents map: pendingDep -> [phases that depend on it]
  - Wave 1 = all phases with inDegree 0
  - For each wave, decrement dependents' inDegree; add to next wave when inDegree reaches 0
  - After processing: if any phases remain with inDegree > 0, `throw new Error("Dependency cycle detected in ROADMAP.md: phases [list] form a cycle")`
- Return array of waves (each wave is PhaseNode[])

**6. `export async function checkWaveConflicts(wavePhases: PhaseNode[], gitRoot: string): Promise<PhaseNode[][]>`**
- For each pair of phases in the wave, call `brokerFetch("/conflict-check", { files: phaseA.filesModified, repo: gitRoot })` — but this checks against running tasks, not other wave phases
- BETTER approach: build a local file-overlap matrix. For each pair (i, j), check if `phaseI.filesModified` and `phaseJ.filesModified` share any paths
- If no conflicts: return `[wavePhases]` (single sub-wave)
- If conflicts detected: use greedy coloring — sort phases by conflict count descending; assign to first sub-wave with no conflict; create new sub-wave if needed
- Return array of sub-waves (each sub-wave's phases have no mutual file conflicts)
- NOTE: Do NOT call broker /conflict-check here — that's for runtime check against already-running tasks. This function handles STATIC planning-time conflicts within a wave.
